## Task Executor

**Назначение:** Исполнитель задач на основе файлов в папке `tasks/` с многослойной структурой (L0–L8).

### Когда использовать

- Для управления разработкой через файлы с задачами
- Когда пользователь говорит: "следующая задача", "запусти задачу", "что дальше", "сгенерируй промпт", "начни разработку", "выполни задачу", "задача готова", "отметь выполненной", "покажи прогресс"
- Для работы с каталогизированными задачами без внешних таск-менеджеров

### Структура папок

```
tasks/
├── 00-guide.md           ← Руководство для LLM — ЧИТАТЬ ПЕРВЫМ
├── _progress.json        ← Создаётся автоматически, отслеживает статус
├── L0/                   ← Фундамент (инициализация, конфигурация)
│   ├── 01-init-project.md
│   └── 02-env-config.md
├── L1/                   ← Слой данных (модели, схемы)
├── L2/                   ← Бизнес-логика
├── L3/                   ← API / Роуты
├── L4/                   ← Аутентификация
├── L5/                   ← Интеграции
├── L6/                   ← Валидация и ошибки
├── L7/                   ← Тесты
└── L8/                   ← Документация и деплой
```

### Примеры команд

| Команда | Режим | Описание |
|---------|-------|----------|
| `следующая задача` | NEXT | Показывает следующую готовую к выполнению задачу |
| `что дальше` | NEXT | То же самое, что и выше |
| `сгенерируй промпт` | PROMPT | Генерирует LLM-промпт для текущей задачи |
| `задача готова` | DONE | Отмечает задачу выполненной |
| `отметь выполненной` | DONE | То же самое |
| `покажи прогресс` | PROGRESS | Показывает текущий статус всех задач |

### Пример формата задачи

```markdown
# L0/01-init-project.md

# Инициализация проекта

**Цель:** Создать базовую структуру проекта с конфигурацией окружения

**Зависимости:** []

**Усилие:** S

**Описание:**
- Создать package.json с базовыми зависимостями
- Настроить ESLint и Prettier
- Создать структуру папок src/

**Критерии завершения:**
- [ ] package.json существует
- [ ] npm install проходит без ошибок
- [ ] Структура папок соответствует требованиям
```

### Ключевые особенности

1. **Всегда читает `00-guide.md`** — содержит правила выполнения для LLM
2. **Отслеживание зависимостей** — задачи блокируются до завершения зависимостей
3. **Статусы задач:** READY, BLOCKED, IN_PROGRESS, DONE
4. **Автоматическое разблокирование** — при завершении задачи разблокируются зависимые

## Brainstorming Skill

**Назначение:** Генерация идей, мозговой штурм и структурированное исследование подходов к решению проблем.

### Когда использовать

- Для генерации идей и креативных решений
- При планировании контента, продуктовых функций, маркетинговых кампаний
- При стратегическом планировании, креативном письме
- Для структурированной генерации идей в любой области

### Рабочий процесс

1. **Изучить контекст** — проверить состояние проекта, файлы, документацию
2. **Задать уточняющие вопросы** — по одному, понять цели/ограничения/критерии успеха
3. **Предложить 2-3 подхода** — с компромиссами и рекомендацией
4. **Представить дизайн** — секциями по масштабу, получить одобрение после каждой
5. **Документировать результат** — записать утверждённый дизайн в markdown-файл
6. **Переход** — вызвать навык планирования или реализации только после одобрения

### Принципы общения

- **Один вопрос за раз** — не перегружайте множественными вопросами
- **Множественный выбор предпочтительнее** — легче ответить, чем открытым вопросом
- **YAGNI безжалостно** — удалять ненужные функции из всех дизайнов
- **Исследовать альтернативы** — всегда предлагать 2-3 подхода
- **Инкрементальная валидация** — представить дизайн, получить одобрение, затем двигаться дальше

### Категории паттернов генерации идей

1. **Умножение перспектив** — генерация идей с разных точек зрения
2. **Вариация ограничений** — исследование пространства идей через искусственные ограничения
3. **Инверсия и отрицательное пространство** — обратное мышление для новых решений
4. **Аналоговый перенос** — применение паттернов из разных областей
5. **Систематическая декомпозиция функций** — SCAMPER и атрибутная идеация
6. **Исследование сценариев** — футуристическое мышление "что если"
7. **Структурированная идеация на основе ограничений** — построение в жёстких ограничениях
8. **Рассуждение цепочкой** — многоступенчатые процессы уточнения
9. **Комбинация и морфологическое исследование** — принудительные новые комбинации
10. **Проблема предположений** — вопрос предпосылок и инверсия допущений
11. **Шаблоны заполнения пропусков** — структурированные форматы завершения
12. **Конкурентное позиционирование** — подходы с матрицей дифференциации
13. **Экстремальное масштабирование** — 10x мышление и экспоненциальные сценарии
14. **Заинтересованные лица и эмпатия** — путешествие клиента и паттерны персон

### Примеры запросов

| Запрос | Паттерн | Результат |
|--------|---------|-----------|
| "Придумай 10 идей для маркетинговой кампании" | Perspective Multiplication | 10 разных идей с точек зрения клиента, конкурента, партнёра |
| "Как мы можем сделать продукт лучше?" | Constraint Variation | Идеи в рамках бюджета, времени, ресурсов |
| "Инновационная функция для приложения" | Inversion + Extreme Scaling | Обратное мышление + 10x масштабирование |

### Важное правило

**HARD-GATE:** Не вызывать никакой навык реализации, не писать код, не создавать структуру проекта и не предпринимать никаких действий по реализации до завершения процесса мозгового штурма, представления дизайна и получения одобрения пользователя.

## Project Portability

**Назначение:** Анализ любого программного проекта и создание комплексного документа портативности (PORT-DOC), который позволяет эффективно пересоздать проект для терминала/CLI, веба, мобильных, десктопа, API или любой другой целевой платформы.

### Когда использовать

- Для портирования или переписывания проекта на другую платформу
- Когда нужна документация основной логики проекта для платформенно-независимого понимания
- При запросе "анализировать проект для веб/мобильной/терминальной версии"
- При фразах типа "документируй это, чтобы я мог пересоздать", "хочу терминальную версию этого", "помоги портировать это на мобильный"
- Даже при простом "анализируй мой проект" или "напиши документацию для портирования"

### Рабочий процесс

```
1. SCAN      → Понять структуру проекта и стек технологий
2. ANALYZE   → Извлечь основную логику, функции, модели данных
3. WRITE     → Создать PORT-DOC (платформенно-независимая спецификация)
4. BREAKDOWN → Генерировать список задач для конкретной платформы
```

### Примеры команд

| Команда | Действие |
|---------|----------|
| "Проанализируй мой проект для портинга на веб" | Анализ + PORT-DOC + веб-задачи |
| "Нужна документация для пересоздания на React Native" | Анализ + PORT-DOC + мобильные задачи |
| "Документируй это для терминальной версии" | Анализ + PORT-DOC + CLI задачи |
| "Создай задачи для rebuild на Python" | Анализ + PORT-DOC + Python задачи |

### Что анализируется

1. **Инвентарь функций** — все пользовательские функции
2. **Модели данных** — сущности, поля, типы, отношения
3. **Основная бизнес-логика** — алгоритмы, правила, вычисления
4. **Ввод/вывод и внешние зависимости** — базы данных, API, файловая система
5. **Потоки пользователей** — основные пути пользователя

### Выходные файлы

1. **`PORT-DOC.md`** — Платформенно-независимая спецификация
2. **`TASKS-[PLATFORM].md`** — Разбивка задач для целевой платформы (если запрошено)

### Пример структуры PORT-DOC

```markdown
# PORT-DOC: Название проекта

## Обзор проекта
[Краткое описание проекта]

## Функции
- [ ] Функция 1: Описание
- [ ] Функция 2: Описание

## Модели данных
### User
- id: UUID
- email: String
- passwordHash: String

## Бизнес-логика
[Описание алгоритмов и правил]

## Потоки пользователей
[Описание основных путей]
```

### Качественные чеклисты

- [ ] Каждая функция из оригинального проекта документирована
- [ ] Модели данных полны со всеми полями
- [ ] Бизнес-логика описана простым языком (без платформо-зависимого кода)
- [ ] Внешние зависимости отмечены с предложенными альтернативами
- [ ] Потоки пользователей охватывают все основные пути
- [ ] Разработчик, незнакомый с оригинальным проектом, может построить новую версию только по этому документу

## Design to Plan

**Назначение:** Декомпозиция любого документа дизайна приложения или API в детальный, понятный для LLM пошаговый план реализации.

### Когда использовать

- Когда пользователь предоставляет документ дизайна (API дизайн, архитектура приложения, системная спецификация, PRD, техническая спецификация, ERD или любой документ планирования)
- При запросе разбить на задачи, подзадачи, дорожную карту, шаги реализации, план спринта или чеклист выполнения
- При фразах "разбить на задачи", "составить план реализации", "пошаговый план", "декомпозировать документ"

### Ключевой принцип: Атомарные шаги

Каждый шаг должен быть:
- **Самодостаточным:** завершаемым без чтения других шагов
- **Проверяемым:** имеет чёткое условие завершения
- **Правильного размера:** 1–4 часа работы разработчика или 1 сфокусированный LLM-промпт
- **Осведомлённым о зависимостях:** упорядочен так, что каждый шаг строится на предыдущем

### Карта слоёв

```
Layer 0: Foundation        (настройка проекта, конфигурация, скелет CI/CD)
Layer 1: Data Layer        (схема БД, модели, миграции, сидеры)
Layer 2: Core Business     (сервисы, доменная логика, вычисления)
Layer 3: API / Interface   (роуты, контроллеры, DTO запросов/ответов)
Layer 4: Auth & Security   (аутентификация, авторизация, мидлвар)
Layer 5: Integration       (внешние API, очереди, вебхуки, email)
Layer 6: Validation & Errors (валидация ввода, коды ошибок, логирование)
Layer 7: Tests             (unit, integration, e2e для каждой функции)
Layer 8: Docs & Deployment (документация API, README, скрипты деплоя)
```

### Формат задачи

```markdown
### L0-01 — Инициализация проекта и конфигурация окружения

**Цель:** Создать базовую структуру проекта с конфигурацией окружения

**Вход:** Файлы конфигурации, список зависимостей

**Выход:** Инициализированный проект с package.json и структурой папок

**Выполнено когда:** npm install проходит без ошибок и структура папок создана

**Подсказка для LLM:** Создай проект React с Vite, добавь зависимости: React, TypeScript, Tailwind

**Оценка усилий:** S (1 час)
```

### Примеры запросов

| Запрос | Действие |
|--------|----------|
| "Разбей этот API дизайн на задачи" | Создаёт план на основе OpenAPI/REST спецификации |
| "Составь план реализации для мобильного приложения" | Декомпозирует по экранам/компонентам |
| "Как реализовать эту архитектуру микросервисов?" | Создаёт план по сервисам с межсервисными контрактами |
| "Декомпозируй эту ERD диаграмму" | Создаёт план: схема → модель → миграция → сервис → тест |

### Специальная обработка

#### Для документов API дизайна
- Группировать эндпоинты по ресурсу (одна группа задач на ресурс)
- Порядок: схема → сервис → роут → валидация → тест
- Отмечать нетривиальные бизнес-правила как COMPLEX

#### Для дизайна мобильных/фронтенд приложений
- Одна задача на экран или основной компонент
- Разделять: настройка state management, навигация, API интеграция, полировка UI

#### Для архитектуры микросервисов
- Одна карта слоёв на сервис
- Добавлять задачи межсервисных контрактов (сначала определить интерфейсы)
- Отмечать общие библиотеки как SHARED — реализовывать до потребителей

### Проверки качества

- [ ] Никакая задача не требует знания из будущего (ещё не выполненного) шага
- [ ] Каждая сущность в документе дизайна имеет хотя бы одну задачу
- [ ] Auth/security не оставлены на конец (Layer 4, а не последняя мысль)
- [ ] Каждая задача имеет измеримое условие завершения
- [ ] Полный план может быть выполнен сверху вниз без циклических зависимостей
- [ ] Подсказки для LLM конкретны, а не абстрактны

## Дополнительные ресурсы

- **Brainstorming Skill:** Библиотека из 30+ паттернов, применения по доменам
- **Project Portability:** Шаблон PORT-DOC, руководства по платформам
